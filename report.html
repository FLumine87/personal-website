1. Introduction

This report documents the development of a real-time multiplayer puzzle game web application for the JC2503 Web Application Development coursework. The project aims to demonstrate proficiency in full-stack web development using modern technologies including Node.js, Express.js, Socket.IO, and EJS templating.

The application implements a turn-based 4×4 grid puzzle game where players take turns placing blocks with unique shapes and colors. The game features real-time multiplayer functionality, automatic line-clearing scoring, time-based turns, and an authoritative server architecture to ensure fair and consistent gameplay across all clients.

The following sections detail the website structure, game design, client-server communication mechanisms, technical challenges encountered during development, and the lessons learned throughout the process.

2. Website Structure

2.1 Page Architecture

The website consists of three main pages: Home, About, and Game, all served through Express.js with EJS templates. The Home page serves as the entry point, featuring a brief introduction to the puzzle game and navigation links. The About page provides detailed information about the game rules, objectives, and technical overview. The Game page contains the core gameplay interface with the 4×4 grid, player information panel, and real-time status updates.

A shared navigation bar is implemented across all pages using EJS partials, ensuring consistent user experience. The responsive design approach uses CSS flexbox and media queries to adapt to different screen sizes, from desktop monitors to mobile devices. The three-page structure was chosen to separate concerns: presentation (Home), documentation (About), and interaction (Game), making the application modular and maintainable.

3. Game Design

3.1 Game Rules and Mechanics

The puzzle game is played on a 4×4 grid with 16 unique blocks, each defined by one of four shapes (circle, square, triangle, diamond) and one of four colors (red, blue, green, yellow). Players take turns drawing a random block from the available pool and placing it on the grid.

When a player places a block, the line-clearing algorithm checks for three or more consecutive matching blocks (by shape or color) in horizontal, vertical, or diagonal directions. Matching lines are cleared, and the player scores points equal to the number of blocks cleared. If all 16 blocks are placed without completing any lines, the player with the highest score wins. A jackpot mechanism rewards players who clear multiple lines simultaneously with bonus points. Each turn has a 60-second timeout to prevent players from stalling the game.

3.2 Authoritative Server Architecture

All game logic is implemented server-side in gameLogic.js following the authoritative server pattern. This design ensures the server maintains the single source of truth for the game state, preventing clients from tampering with game data or implementing cheating mechanisms.

Security Implications: By keeping game logic server-side, clients cannot modify critical data such as scores, block positions, or turn order. All client actions (placing blocks) are validated by the server before being processed, ensuring only valid moves are accepted.

Multiplayer Consistency: The authoritative server guarantees that all players see the same game state at any given time. When a player places a block, the server processes the request, updates the state, and broadcasts changes to all connected clients. This eliminates synchronization issues that would arise if each client maintained its own state.

3.3 Key Algorithms

Line-Clearing Algorithm

The line-clearing algorithm detects three or more consecutive blocks with matching shape or color by searching in eight directions from the placed block position. The algorithm uses four direction vectors: horizontal (left/right), vertical (up/down), and two diagonals. For each direction pair, it extends outwards from the placed block, collecting indices of matching blocks until a mismatch is encountered.

A Set data structure is used to collect all matching block indices, automatically eliminating duplicates when a block belongs to multiple intersecting lines. After identifying all matches, the algorithm removes those blocks from the grid, shifts remaining blocks down (if applicable), and calculates the player's score based on the number of blocks cleared.

Turn Management and Timeout

The server manages turns using a round-robin approach, cycling through connected players in the order they joined. Each turn has a 60-second timeout implemented with JavaScript's setTimeout. When a timeout occurs, the server skips the current player's turn, removes their assigned block from the available pool, and moves to the next player.

Proper cleanup is implemented using clearTimeout when a player disconnects or the game ends, preventing orphaned timeout callbacks. If all players disconnect, the game state is reset for future players.

4. Client-Server Communication

4.1 Socket.IO Architecture

Socket.IO enables real-time multiplayer functionality by establishing persistent WebSocket connections between clients and server. This architecture provides instant bidirectional communication, essential for maintaining synchronized game state across all players.

Client-to-Server Events:

  • joinGame: Emitted when a player enters the game page with their username
  • placeBlock: Sent when a player clicks a grid cell to place their block
  • leaveGame: Triggered when a player leaves the game

Server-to-Client Events:

  • gameStateUpdate: Broadcasts the complete game state (board, players, scores) to all clients
  • yourTurn: Informs a specific player it's their turn and sends their assigned block
  • boardUpdate: Sends updated grid state after a block placement
  • turnChange: Notifies all players when the turn switches to another player
  • gameEnd: Announces the game result when all blocks are placed

This event-driven architecture is suitable for multiplayer games because it ensures all players receive immediate updates without polling, reducing latency and improving user experience.

5. Challenges & Lessons

5.1 Technical Challenges

Challenge 1: Line-Clearing Algorithm Complexity

Implementing the line-clearing logic was challenging due to the need to check all four directions (horizontal, vertical, and two diagonals) for both shape and color matches. The complexity increased when handling cases where multiple lines intersect at the same block, potentially causing duplicate removals or incorrect scoring.

Solution: I implemented a directional search approach that extends from the placed block position in positive and negative directions for each axis. The algorithm collects matching block indices using a Set data structure, which automatically handles duplicates. After identifying all matches, the Set is converted to an array, and blocks are removed from the grid simultaneously. This ensures each block is only counted once, even if it belongs to multiple lines.

Challenge 2: Multiplayer State Synchronization

Ensuring all players see the same game state in real-time was particularly challenging when players joined or left mid-game. Initial attempts resulted in desynchronization, where new players would see an incomplete board or incorrect turn information.

Solution: I adopted the authoritative server pattern, where all game logic runs exclusively on the server. When a new player connects, the server immediately sends the complete current game state via the gameStateUpdate event. After every action (block placement, line clearing, turn change), the server broadcasts updates to all connected clients, ensuring everyone stays synchronized.

Challenge 3: Timeout Management with Dynamic Player List

Managing 60-second timeouts became complicated when players dynamically joined or left the game. The initial implementation had issues with orphaned timeout callbacks and incorrect turn progression when a player disconnected during their turn.

Solution: I implemented robust timeout management using JavaScript's setTimeout with corresponding clearTimeout calls for cleanup. When a player disconnects, any pending timeout for their turn is immediately cleared. The server then recalculates the turn order and starts a new timeout for the next player. This prevents orphaned callbacks from firing and ensures smooth turn progression regardless of player changes.
